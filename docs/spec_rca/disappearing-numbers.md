## 1. Root Cause

`TimerComponent` is a CDN component whose constructor takes the container element ID as a **first positional string argument**: `new TimerComponent('timer-container', { options })`. Build #442's base generated HTML calls it incorrectly using an options object with a DOM element reference as the first argument:

```javascript
// WRONG (build 442):
timer = new TimerComponent({
  container: document.getElementById('timer-container'),
  timerType: 'increase', startTime: 0, endTime: 3600, autoStart: false, format: 'min'
});
```

When `TimerComponent` receives an object as its first argument, it calls `document.getElementById({...})`, which coerces the object to `"[object Object]"`, finds nothing, and throws `Container with id "[object Object]" not found`. This exception is caught by the DOMContentLoaded try/catch, sets `window.__initError`, and halts all further initialization — `ProgressBarComponent`, `TransitionScreenComponent`, and `transitionScreen.show()` are never reached. The `#mathai-transition-slot` exists in the DOM (ScreenLayout.inject succeeds) but never receives a start button. The test's `beforeEach` loop times out at 50 seconds.

This is **not a CDN loading race condition**. All 7 CDN packages load successfully within 500ms. The failure is a wrong API call signature generated by the LLM.

**Secondary issue in pipeline's mid-run fix attempt (index-global-fix1.html):** A prior diagnostic session (this file's v1) ran against `index-global-fix1.html` from GCP — a version where the LLM had already corrected the `TimerComponent` call but then introduced `new TransitionScreenComponent({ containerId: 'mathai-transition-slot' })` (wrong API) instead of `new TransitionScreenComponent({ autoInject: true })`. That fix attempt was mid-pipeline when the worker was restarted (build orphaned). The secondary issue is real but only appears in the LLM's attempted repair, not the base HTML.

## 3. POC Fix Verification

Applied fix locally by patching the HTML in-memory before serving it via HTTP and running Playwright assertions:

**Fix applied:**
```javascript

timer = new TimerComponent({
  container: document.getElementById('timer-container'),
  timerType: 'increase', startTime: 0, endTime: 3600, autoStart: false, format: 'min'

// After (correct):
timer = new TimerComponent('timer-container', {
  timerType: 'increase', startTime: 0, endTime: 3600, autoStart: false, format: 'min'
});
```

**POC results (node script run against patched HTML):**
```
__initError after fix: undefined                      ← No crash

gameState.phase: start_screen                         ← Game initialized correctly
```

Without fix: `__initError` = `"Container with id \"[object Object]\" not found"`, transition slot button = false, phase = `idle`.

The fix is a 1-line constructor call change. No pipeline or test code changes needed — this is purely a gen prompt issue.

## 4. Reliability Reasoning

**Is the fix deterministic?** Yes, once the correct call is generated. `TimerComponent('timer-container', opts)` is the documented CDN API, confirmed stable across 4+ approved builds.

**What could cause regression?**
1. LLM re-generates the wrong API call — likely since build 442 got it wrong despite having the correct pattern in context. The gen prompt must add an explicit, prominent rule: `TimerComponent(containerId: string, options: object)` — first argument is a string ID, NOT `document.getElementById(...)`, NOT an options object.
2. If a future build corrects `TimerComponent` but also generates `new TransitionScreenComponent({ containerId: ... })` (as the LLM's mid-run fix did), that will produce the secondary error. The gen prompt must also specify: `new TransitionScreenComponent({ autoInject: true })`.
3. Worker restart could orphan the build again (infra issue, unrelated to HTML correctness).

**Edge cases handled:**
- `timer-container` element exists in the HTML's `<template>` tag which is appended to `#gameContent` after `ScreenLayout.inject()`. The `TimerComponent` constructor call happens after that template clone, so `#timer-container` exists at call time.
- Audio preload URLs 404 locally (no `test-dynamic-assets` server) but are non-fatal — caught in try/catch.
- `FeedbackManager.init()` is correctly omitted (not called) — no audio popup issue.

**Why this fix holds across multiple builds:** The CDN API is stable (not versioned in URLs). Approved games prove `TimerComponent('id', opts)` works reliably. The root cause is purely in how the LLM generates the constructor call.

## Build 400 Notes (Separate Issue)

Build #400 had a different failure: `"Step 1d: Page load failed: Blank page: missing #gameContent element"`. That generation used `ScreenLayout.inject('app', { progressBar: true, transitionScreen: true })` — **without** the required `slots:` wrapper key. Without `{ slots: { ... } }`, `ScreenLayout.inject()` does not create `#gameContent`, `#mathai-transition-slot`, or `#mathai-progress-slot`. The subsequent `document.getElementById('gameContent').appendChild(...)` call throws a null dereference, the catch block logs `Init error: {}` (empty object, no `.message` property), and the page stays blank. The T1 static validator catches this as "missing `#gameContent`" at Step 1d.

This is a distinct bug from build 442's `TimerComponent` issue, but may share a common cause: the LLM not reliably generating the correct CDN component API calls.

## Targeted Fix Summary

No targeted fix applied yet. All failures were either infra (builds 67–280, 442) or caught at Step 1d before tests ran (build 400). The correct fix for the next build is:

1. **Gen prompt must specify:** `TimerComponent(containerId: string, options: object)` — first arg is string ID, second arg is options object. NOT `{container: element, ...}` as single argument.
2. **Gen prompt must specify:** `new TransitionScreenComponent({ autoInject: true })` — not `containerId`.
3. **Gen prompt must specify:** `ScreenLayout.inject('app', { slots: { progressBar: true, transitionScreen: true } })` — must have `slots:` wrapper (issue seen in build 400).
4. **No pipeline code changes needed** — these are all LLM generation errors, not pipeline logic errors.
