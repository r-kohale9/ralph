## Methodology

The linter (`lintGeneratedTests()` in `lib/pipeline-test-gen.js`) was committed on 2026-03-22. All 664 spec files on the server predate it — they were generated by the LLM before the linter existed. This gives a clean baseline of how often each rule would have fired on real-world LLM output.

Two analyses were run:

1. **Full corpus:** 664 spec files across 145 build directories (all builds ever)
2. **Recent builds only:** 52 spec files from 11 approved builds (IDs 505–515, all from 2026-03-21), representing the most current LLM generation quality

All scripts were run directly on the production server against `/opt/ralph/data/games/`.

## Rule-by-Rule Findings

### TRANSITION_SLOT — Critical (fires on 99% of all files)
The pattern `#mathai-transition-slot button` appears in virtually every generated spec file because LLMs consistently use it to advance between game phases. The rule correctly identifies that `type=level-transition` auto-advances and should never be waited on or clicked in tests. This is the highest-value rule — it catches a universal anti-pattern.

**Sample violation:**
```js
await page.locator('#mathai-transition-slot button').first().click();
```

### GF8 — Very High (fires on 47% of all files, 64% of recent builds)
`toBeVisible()` assertions are used without a preceding `waitForPhase()` call. This is a race condition pattern — the element may not be visible because the phase transition hasn't completed. Common in results-screen and game-element visibility checks.

**Sample violations:**
```js
await expect(page.locator("#results-screen")).toBeVisible();
await expect(page.locator('#number-list-area')).toBeVisible();
```

### M13 — High (fires on 19% of all files, 45% of recent builds)
Immediate `getLives(page)` / `getScore(page)` / `getRound(page)` assertions without `expect.poll()` are a classic flaky-test source — the DOM hasn't updated yet when the assertion runs. This remains common even in the most recent builds, indicating the LLM gen prompt hasn't fully absorbed the lesson.

**Sample violations:**
```js
expect(await getLives(page)).toBe(3);
expect(await getScore(page)).toBe(0);
```

### RULE-DUP — Moderate (fires on 6.9% of all builds, 27% of recent builds)
Duplicate `data-testid` values across spec files in the same build. The higher rate in recent builds (27% vs 6.9% overall) may indicate this is actually getting worse, or it could be sampling noise. Worth monitoring.

**Sample violations (build 514, match-the-cards):**
- `data-testid="left-card-0"` in both `contract.spec.js` and `edge-cases.spec.js`

### M15 — Never fires (0 hits across 664 files)
The pattern checks for `.toHaveClass('correct'/'wrong'/'disabled'/'active')` used without `expect.poll()`. The LLM apparently doesn't use `.toHaveClass()` with these specific class names — it uses other assertion styles. The rule is well-intentioned but the pattern does not match real-world LLM output.

### CT7 — Never fires (0 hits)
The pattern `msg.type === 'gameOver'` was intended to catch the wrong postMessage type. The LLM has apparently learned to use `'game_complete'` correctly, or uses different code patterns for postMessage checks. Not a current problem.

### CT6_NULL — Never fires (0 hits)
The multiline pattern for `getLastPostMessage()` without a null-guard doesn't match any real test. Either the LLM always null-guards, or the pattern is too narrow to catch the actual code shapes produced. The multiline regex (`\n`) may not match cross-platform line endings.

### CT7_GAMEOVER — Never fires (0 hits)
`waitForPhase(page, 'game_over')` never appears — the LLM uses `'gameover'` (normalized form) correctly, or has internalized the right phase names.

---

## Conclusion: Linter Effectiveness

**3 of 8 rules are high-value and fire regularly:**
- `TRANSITION_SLOT` (nearly universal — 99% of files)
- `GF8` (common — 47% of files)
- `M13` (common — 19% of files)

**1 rule fires occasionally and adds real value:**
- `RULE-DUP` (6.9% of builds, 27% recent)

**4 of 8 rules never fire and need rethinking:**
- `M15`, `CT7`, `CT6_NULL`, `CT7_GAMEOVER` — 0 hits across 664 files

---

## Recommendations

### 1. Rules to keep as-is (high value)
- `TRANSITION_SLOT`, `GF8`, `M13`, `RULE-DUP` — all confirmed catching real patterns

### 2. Rules to refine or replace (0 hits)

**M15:** Widen pattern. LLMs use `expect(el).not.toHaveClass(...)` or check CSS classes via `getAttribute('class')`. Suggested replacement:
```js
/\.toHaveClass\s*\(/ // any .toHaveClass() without poll, not just state classes
```

**CT7:** Retire or replace. `'gameOver'` as a wrong type is not in current LLM output. More relevant today might be `msg.type === 'score_update'` or other incorrect types seen in recent failures.

**CT6_NULL:** Fix the multiline regex. The `\n` in the pattern won't match when line-by-line scan is used (the code does try a full-content match for patterns with `\n`, but the regex may still not match due to whitespace variations). Simplify to single-line: check for `getLastPostMessage(page)` assigned to a variable, then used without `if (msg)` nearby.

**CT7_GAMEOVER:** Retire or broaden. The LLM is already using the normalized phase names. Replace with a check for other unnormalized phase names: `'game_complete'` vs `'results'`, `'start_screen'` vs `'start'`.

### 3. Suggested new rules based on corpus analysis

After reviewing sample violations, two additional patterns are widespread and not currently caught:

**NEW-A — page.locator without waitFor:** Many tests use `page.locator(...).click()` directly without any `waitFor*` call, causing flaky failures when elements are not yet interactive.

**NEW-B — hardcoded timing:** `page.waitForTimeout(N)` appears in many tests. This is a known flakiness source on slow server Playwright.

### Additional false positive found (not in the 8 listed — still unfixed)

The **`endGame` async check** at `validate-static.js` line 140 has a regex scope bug:

```js
const fnMatch = html.match(new RegExp(`((?:async\\s+)?function\\s+${fnName}\\s*\\([^)]*\\)\\s*\\{)([\\s\\S]*?)(?=\\nfunction |\\n(?:const|let|var)\\s+\\w|\\n\\/\\/\\s*─|$)`));
```

The lookahead `(?=\nfunction )` requires `function` at column 0. When game code is indented (8-space indent inside `DOMContentLoaded`), the pattern `\n        function` never matches the lookahead, so the regex captures ALL code from `endGame` onwards — including the body of `restartGame` and other later functions. In build 531, `endGame` had no `await` but `restartGame` (immediately following) did. The check fired as a false positive.

**Build 531 observed:** T1 fired `endGame() uses await but is not declared async` — `endGame` function at line 1016 has no `await`; `restartGame` at line 1073 is `async function` with `await transitionScreen.show()`, but that body was captured as part of `endGame`'s match. This is NOT fixed by db38ede (which fixed the arrow-fn declaration check, a different issue).

### Checks with 0 true positives across all 8 builds

| Check | Reason for 0 TP | Action |
|-------|----------------|--------|
| §5i RULE-SYNC-1 | LLM never put `setTimeout` inside `startGame()` in these builds (confirmed by build 535 manual check) | **keep** — gen prompt is working; check is fast insurance |
| §5k CDN namespace | LLM used `window.mira` (caught by §5j), not `window.cdn` etc. | **keep** — complements §5j |
| §5l require/import | LLM uses `<script>` tags consistently for CDN packages | **keep** — cheap guard |

### Open issue: `endGame` async scope bug (line 140, still unfixed)

The regex lookahead assumes un-indented function declarations. For any game where functions are indented inside `DOMContentLoaded`, the scope extends to EOF, producing false positives. Fix: use brace-counting (like the §5i check does at line 670+) instead of regex lookahead. Priority: medium — only fires when `endGame` is sync and a later function is async.
